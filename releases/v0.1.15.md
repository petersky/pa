# v0.1.15 — 0.1.15

**Release track:** release  
**Date:** 2026-07-14

## Summary

This release makes fleet readiness warnings actionable from the `/fleet` wizard. Fleet operators who saw reachability blockers in v0.1.14—loopback URLs, local-only bind, or a missing sync token—can now fix them in the UI without editing config or running CLI commands.

## Highlights

- Fleet readiness issues show how-to-fix text and inline controls on `/fleet`
- Set advertised URL, bind to 0.0.0.0, and generate a sync token from the readiness panel
- `POST /api/fleet/readiness` persists reachability settings and restarts PA when bind host changes
- Loopback advertised URLs are rejected with clear validation errors

## Added

- Structured readiness issues with fix hints and per-issue UI actions (`set_instance_url`, `set_bind_all`, `ensure_sync_token`)
- `POST /api/fleet/readiness` to save advertised URL and bind host from the Fleet UI
- Reachability settings panel on `/fleet` for combined URL and bind-host edits
- `apply_reachability_settings` helper with validation and service env refresh
- `host` field on instance config for persisted bind-address settings
- Unit tests for reachability persistence and loopback URL rejection

## Changed

- Fleet readiness warnings are surfaced as actionable issues instead of plain text
- Changing bind host rewrites the service unit and triggers an automatic restart so peers can connect

## Fixed

- Fleet readiness warnings were informational only—users could not resolve reachability blockers from the UI
